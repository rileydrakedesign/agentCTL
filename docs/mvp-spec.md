# agentctl MVP Spec

## Context

Developers using Claude Code (and similar AI agents) configure MCP servers, skills, and prompts without visibility into how those configurations impact context window usage, cost, or capability overlap. agentctl is a CLI-first configuration intelligence tool that connects to live MCP servers, analyzes the full capability stack, and produces actionable simulation reports — before the agent ever runs.

This MVP targets **power users / early adopters** who already have complex MCP setups and feel the pain of context bloat, redundant tools, and opaque cost.

---

## MVP Scope

### What's IN
- `agentctl init` — scaffold project config
- `agentctl scan` — live MCP discovery via `tools/list`, skill parsing
- `agentctl plan` — token simulation, capability analysis, recommendations
- `agentctl doctor` — validate config health, connectivity, auth
- Workspace analysis — aggregate metrics across all MCP servers in a config
- Terminal output (human-readable) + JSON output (`--json`)
- Artifact output to `.agentctl/` directory

### What's OUT (post-MVP)
- Dashboard / web UI
- `agentctl diff` (revision comparison)
- `agentctl push` (cloud upload)
- `agentctl policy` (enforcement rules)
- Telemetry / runtime trace ingestion
- Multi-platform config support (Cursor, Windsurf, etc.)
- Strategy engine (workspace restructuring recommendations)
- Cost-per-outcome / economics engine

## Input Contract

### Primary input: Claude Code MCP config
Supported file locations (auto-discovered in order):
1. `.mcp.json` (project-level Claude Code config)
2. `~/.claude/claude_desktop_config.json` (user-level)
3. Explicit path via `--config <path>`

#### Config shape (what we parse):
```json
{
  "mcpServers": {
    "<server-name>": {
      "command": "string",
      "args": ["string"],
      "env": { "KEY": "value" },
      "url": "string (for SSE/streamable HTTP transport)"
    }
  }
}
```

### Secondary inputs (auto-discovered):
- `CLAUDE.md` / `AGENTS.md` — prompt files, estimated for token cost
- `skills/` or `.skills/` directories — parsed for SKILL.md metadata
- `.env` — checked for referenced env var availability

### Project config (generated by `init`):
```yaml
# agentctl.yaml
version: 1
project:
  name: "my-agent"
runtime_targets:
  - model: claude-sonnet-4-6
    context_window: 200000
  - model: gpt-4o
    context_window: 128000
```

## Discovery Model (Live)

### Process per MCP server:
1. Resolve transport (stdio command or SSE/HTTP URL)
2. Start server process or connect to endpoint
3. Call `tools/list` via MCP protocol
4. Collect: tool name, description, input schema (JSON Schema)
5. Record: success/failure, latency, tool count
6. Shut down / disconnect

### Failure handling:
- Missing env vars → diagnostic: `auth_failure`, server marked excluded
- Server won't start → diagnostic: `connectivity_failure`
- `tools/list` times out (default 30s) → diagnostic: `timeout`
- Partial results → compile status: `PARTIAL`
- All servers fail → compile status: `FAILED`

### Output per server:
```json
{
  "server": "github",
  "status": "ok | error",
  "transport": "stdio | sse",
  "tools": [
    {
      "name": "create_issue",
      "description": "Create a new GitHub issue...",
      "input_schema": { ... },
      "token_estimate": {
        "description_tokens": 45,
        "schema_tokens": 120,
        "total": 165
      }
    }
  ],
  "diagnostics": []
}
```

## Token Estimation

### What gets counted:
1. **Tool descriptions** — tokenized text
2. **Tool input schemas** — serialized JSON Schema, tokenized
3. **Prompt files** — CLAUDE.md, AGENTS.md content
4. **Skill instructions** — SKILL.md content

### Tokenizer:
- Use `tiktoken` (cl100k_base) as baseline tokenizer
- Apply model-specific scaling factors for budget projections:
  - Claude models: 1.0x (native estimate)
  - GPT models: 1.0x (tiktoken native)
  - Gemini: 1.1x (approximate)

### Budget categories:
- **Discovery tokens**: always loaded into context (all tool descriptions + schemas)
- **Activation tokens**: loaded when a tool is actually invoked (estimated via heuristics)
- **Prompt tokens**: system prompts, CLAUDE.md, etc.
- **Total projected**: discovery + estimated activation + prompts

### Activation modeling (MVP-simple):
- Discovery mode: all tool descriptions + schemas loaded
- Typical: 30% of tools activated (configurable)
- Worst-case: 100% of tools activated

## Analysis Engine

### Redundancy detection:
- Compare tool descriptions pairwise using normalized text similarity
- Flag clusters where similarity > 80% threshold
- Example: "search_issues" from GitHub MCP vs "find_jira_tickets" from Jira MCP

### Dead capability detection:
- Tools with empty or near-empty descriptions
- Tools whose schemas reference undefined types
- Servers that returned 0 tools

### Context pressure warnings:
- Discovery tokens > 50% of smallest target context window
- Single MCP server contributing > 40% of total token budget
- Total tool count > 50

## Workspace Analysis

### Metrics computed:
- Total MCP server count
- Total tool count (across all servers)
- Total discovery token cost
- Per-MCP token contribution (cost attribution)
- Redundancy cluster count
- Estimated waste % (redundant + low-value tools / total)

## CLI Commands

### `agentctl init`
- Interactive setup: project name, runtime targets
- Auto-discovers existing MCP configs
- Writes `agentctl.yaml`

### `agentctl scan`
- Connects to all configured MCP servers
- Calls `tools/list`, collects capabilities
- Parses skills directories
- Outputs `scan.json` to `.agentctl/latest/`
- Prints summary to terminal

### `agentctl plan`
- Runs scan (or uses cached scan with `--cached`)
- Computes token budgets per runtime target
- Runs analysis (redundancy, dead caps, context pressure)
- Computes workspace metrics
- Outputs full artifact bundle to `.agentctl/latest/`
- Prints plan report to terminal

Flags:
- `--json` — JSON output to stdout
- `--models <list>` — override runtime targets
- `--out <dir>` — custom output directory
- `--fail-on <level>` — exit non-zero on warnings/errors (for CI)
- `--max-discovery-tokens <n>` — fail if discovery exceeds budget
- `--cached` — use last scan instead of re-scanning

### `agentctl doctor`
- Validates agentctl.yaml
- Checks each MCP server: can it connect? does auth work?
- Checks env var availability
- Checks for referenced files (scripts, skill dirs)
- Prints pass/fail per check

## Development Sequence

1. **Project scaffolding** — repo structure, CLI framework, `init` command
2. **Config parser** — read .mcp.json / claude_desktop_config.json, parse agentctl.yaml
3. **MCP client** — connect to servers (stdio + SSE), call `tools/list`, handle failures
4. **Token engine** — tokenize descriptions + schemas, compute budgets
5. **Analysis engine** — redundancy detection, dead caps, context pressure warnings
6. **Workspace aggregation** — per-MCP cost attribution, waste %, cluster detection
7. **Plan output** — terminal formatter + JSON artifact writer
8. **Doctor command** — config validation + connectivity checks
9. **Tests + polish** — integration tests against mock MCP servers
