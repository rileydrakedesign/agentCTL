# agentctl

CLI-first configuration intelligence for AI agent toolchains. Analyze your MCP server setup before your agent runs — catch context bloat, capability overlap, and dead tools early.

## The Problem

When you wire up multiple MCP servers to an AI agent, you have no visibility into:

- How much of your context window the tool definitions consume
- Whether tools across servers overlap (wasting tokens on duplicates)
- Which tools are broken or effectively dead (empty descriptions, invalid schemas)
- Whether your setup will even fit in the model's context window

**agentctl** answers all of this with a single command.

## Quick Start

```bash
# Install
pnpm install -g agentctl

# Initialize project config
agentctl init --name my-agent

# Run a full analysis
agentctl plan

# Or just discover capabilities
agentctl scan

# Check config health
agentctl doctor
```

## What It Does

agentctl connects to your MCP servers, pulls their tool manifests, and runs a multi-stage analysis pipeline:

```
Discover → Budget → Analyze → Report
```

1. **Discover** — connects to each MCP server via stdio or SSE, calls `tools/list`, records tool metadata
2. **Budget** — tokenizes every tool description and schema using tiktoken, computes per-model context usage
3. **Analyze** — detects redundancy (TF-IDF cosine similarity), dead capabilities, and context pressure
4. **Report** — renders results to terminal, JSON, and file artifacts

## Commands

### `agentctl init`

Scaffold an `agentctl.yaml` project config.

```bash
agentctl init --name my-agent
```

### `agentctl scan`

Discover capabilities from all configured MCP servers.

```bash
agentctl scan                    # Terminal output
agentctl scan --json             # JSON to stdout
agentctl scan --config path.json # Explicit config path
agentctl scan --out ./reports    # Custom artifact directory
```

### `agentctl plan`

Full analysis pipeline — scan, budget, analyze, and report.

```bash
agentctl plan                              # Full run
agentctl plan --cached                     # Reuse last scan
agentctl plan --json                       # JSON output
agentctl plan --models claude-sonnet-4-6,gpt-4o  # Override models
agentctl plan --fail-on warning            # Exit non-zero on warnings
agentctl plan --max-discovery-tokens 50000 # Fail if over budget
```

| Flag | Description |
|------|-------------|
| `--config <path>` | Path to MCP config file |
| `--json` | Output JSON to stdout |
| `--out <dir>` | Custom output directory (default: `.agentctl/latest/`) |
| `--cached` | Use last scan instead of re-scanning |
| `--models <list>` | Comma-separated model list |
| `--fail-on <level>` | Exit non-zero on `warning` or `error` (default: `off`) |
| `--max-discovery-tokens <n>` | Fail if discovery tokens exceed budget |

### `agentctl doctor`

Validate configuration health and MCP server connectivity.

```bash
agentctl doctor
agentctl doctor --config path.json
```

Checks:
- `agentctl.yaml` validity
- MCP config file discovery and parsing
- Environment variable availability (for server auth)
- MCP server connectivity
- Prompt files and skill directories

## Configuration

### MCP Config (auto-discovered)

agentctl looks for MCP server configuration in this order:

1. `.mcp.json` in the project directory
2. `~/.claude/claude_desktop_config.json` (user-level)
3. Explicit path via `--config`

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "$GITHUB_TOKEN" }
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/tmp"]
    }
  }
}
```

### Project Config (`agentctl.yaml`)

Generated by `agentctl init`. Defines runtime targets for budget analysis.

```yaml
version: 1
project:
  name: my-agent
runtime_targets:
  - model: claude-sonnet-4-6
    context_window: 200000
  - model: gpt-4o
    context_window: 128000
```

### Skills

agentctl discovers skills from `skills/` or `.skills/` directories containing `SKILL.md` files:

```markdown
---
name: code-review
description: Review code changes for bugs, style issues, and best practices
---

Review the provided code diff for potential bugs, style violations,
and adherence to best practices.
```

## Output Artifacts

Each run writes artifacts to `.agentctl/latest/`:

| File | Contents |
|------|----------|
| `scan.json` | Raw MCP discovery results (servers, tools, token estimates) |
| `plan.json` | Full analysis report (budgets, workspace metrics, recommendations) |
| `diagnostics.json` | Connection failures, auth errors, timeouts |
| `recommendations.json` | Actionable optimization suggestions |
| `summary.txt` | Plain-text terminal output (ANSI stripped) |

## Analysis Features

### Token Budget Estimation

Uses `js-tiktoken` (cl100k_base encoding) with model-specific scaling:

| Model Family | Scale Factor |
|-------------|-------------|
| Claude | 1.0x |
| GPT | 1.0x |
| Gemini | 1.1x |

Budget categories:
- **Discovery** — tool descriptions + JSON schemas (always loaded by the host)
- **Prompt** — `CLAUDE.md`, `AGENTS.md` content
- **Skill** — `SKILL.md` files
- **Activation** — estimated 30% typical, 100% worst-case

### Redundancy Detection

TF-IDF cosine similarity across tool descriptions. Flags cross-server pairs exceeding 80% similarity, with estimated token savings if consolidated.

### Dead Capability Detection

Flags tools with:
- Empty or near-empty descriptions (< 5 characters)
- Broken JSON Schema references
- Servers returning zero tools

### Context Pressure Warnings

- Discovery tokens > 50% of smallest context window
- Single MCP server > 40% of total discovery tokens
- Tool count > 50

## CI Integration

Use `--fail-on` and `--max-discovery-tokens` to gate CI pipelines:

```bash
# Fail if any warnings are found
agentctl plan --fail-on warning

# Fail if discovery tokens exceed 50k
agentctl plan --max-discovery-tokens 50000

# JSON output for downstream processing
agentctl plan --json > report.json
```

### Project Structure

```
src/
├── cli/           # Command handlers (init, scan, plan, doctor)
├── config/        # Config discovery, parsing, Zod schemas
├── mcp/           # MCP client (stdio + SSE transport)
├── tokens/        # Tiktoken wrapper, budget computation
├── analysis/      # Redundancy, dead caps, pressure detection
├── workspace/     # Aggregate metrics
├── skills/        # SKILL.md parsing
├── output/        # Terminal rendering, JSON, file artifacts
└── types.ts       # Centralized type definitions
```

### Tech Stack

- **TypeScript** (strict, ESM-only) targeting Node 20+
- **Commander** for CLI parsing
- **@modelcontextprotocol/sdk** for MCP communication
- **js-tiktoken** for token counting
- **Zod** for config validation
- **chalk** + **cli-table3** + **ora** for terminal UI
- **vitest** for testing, **tsup** for bundling
